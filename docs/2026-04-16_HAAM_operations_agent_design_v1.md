## 1. 작업 컨텍스트

### 1.1 배경

HAAM(도심창고:함)은 숭실대입구 1개 지점을 운영하는 셀프스토리지다. 현재 운영 시스템은 **GAS + Google Sheets + 텔레그램 봇 + Gemini OCR + Cloudflare Workers 정적 페이지** 조합이며, 유입은 전적으로 네이버 지도 → 네이버 예약에 의존한다. 이를 자체 웹사이트 + 토스페이먼츠 결제로 대체해 네이버 수수료/수동 결제 확인/이중 작업을 제거하고, DB/백엔드도 현대적 스택으로 이관한다.

### 1.2 목적

Claude Code 프로젝트에 배치될 에이전트 시스템이 HAAM 신규 웹사이트의 **설계→구현→마이그레이션→배포** 전 단계를 일관되게 수행하도록 워크플로우·스킬·서브에이전트 경계를 사전에 정의한다. 비즈니스 규칙과 알림톡 템플릿 등 **변경 금지 영역**과 기술적 **대체 영역**을 구분해 회귀 오류를 막는다.

### 1.3 범위

**In-Scope**
- Next.js 기반 고객용 웹사이트(모바일 퍼스트 예약/결제/마이페이지)
- 관리자 대시보드(계약현황/매출/만료예정/수동작업)
- Postgres 스키마 설계 및 Google Sheets → Postgres 데이터 마이그레이션
- 토스페이먼츠 결제 위젯 연동 + 웹훅 처리
- Solapi 알림톡 13개 템플릿 이관 및 이벤트 연계
- 만료/연체 자동화 배치(D-7/D-0/D+1/D+3/D+7)
- Hugen-V3 QR 큐 인터페이스(물리 장비 연동은 유지)

**Out-of-Scope (v1)**
- 고객 문의 챗봇, 이상 감지, 매출 리포트 자동 생성 등 운영 시점 LLM 판단 기능
- 복수 지점 확장 (단, 스키마는 `branch_id` 필드로 확장 여지 확보)
- Hugen-V3 자체 개선 (기존 pywinauto 봇 유지)
- 네이버 예약 병행 운영 (일괄 대체)

### 1.4 입력 / 출력

| 구분 | 내용 |
|------|------|
| **에이전트 입력** | `docs/haam-operations-handoff.md`(도메인 지식), `legacy-gas/*.gs`(비즈니스 로직 원본), 환경변수(Supabase/Toss/Solapi 키), 사용자 승인/결정 |
| **에이전트 출력** | 실행 가능한 Next.js 코드베이스, Drizzle 스키마 + 마이그레이션, 테스트, 마이그레이션 리포트, 배포 산출물 |
| **중간 산출물** | `output/` 하위 JSON/SQL/MD 파일 (Phase 간 전달) |
| **최종 산출물** | 배포된 웹사이트 URL, 관리자 대시보드 URL, 마이그레이션 완료 리포트 |

### 1.5 제약조건

1. **비즈니스 규칙 불변**: 가격 계산(정가/장기할인/프로모션 토글), 만료 타임라인(D-7/D-0/D+3), 연체 에스컬레이션(D+1/D+3/D+7), 보증금 정책(계약시 수납·종료 후 3일 반환), 공동사용자 상한(M2/L3/XL4)은 현행 그대로.
2. **알림톡 변경 최소화**: Solapi 등록 템플릿 13개의 변수 매핑·발송 시점은 현행 유지. 미등록 템플릿(연체 D+1/D+3/D+7)은 신규 등록 필요.
3. **Hugen-V3 연동 유지**: 새 DB에도 QR 등록 대기열 테이블을 두고, 기존 pywinauto 봇이 읽을 수 있는 API/뷰를 제공.
4. **1인 운영 전제**: 배포·모니터링·비용이 한 사람이 감당 가능한 수준.
5. **KST 타임존 일관**: 시작일/만료일/알림 발송 시각은 모두 KST 기준. DB는 UTC 저장 + 표시 계층에서 변환.
6. **보안**: 보안코드 5자리(첫자리 1~9), 보증금 반환 토큰 7일 유효, 공동사용자 초대 토큰 24시간 유효.
7. **결제 원자성**: 결제 성공 웹훅 수신 전에는 계약을 `완료`로 승격하지 않음. 웹훅 재전송 허용 설계.

### 1.6 용어 정의

| 용어 | 정의 |
|------|------|
| Active / Archive | 현행 Google Sheets의 핵심 테이블. 계약 중 / 종료 고객. 새 DB에서는 `contracts` 단일 테이블 + `status` enum으로 통합 |
| RENEWAL 상태 | 계약의 갱신/연체 상태 머신. (빈값→안내완료(D-7)→종료, 연체관리→연체(D+1/D+3/D+7), 재계약) |
| 보안코드 | 5자리 숫자. QR 출입 인증 (Hugen-V3) |
| 뱃지 | 네이버 예약의 상태 라벨. 신규 시스템에서는 `orders.status` 로 대체 |
| 프로모션 토글 | 오픈 기념 할인 On/Off 전역 플래그. On이면 장기할인 대신 무료 개월 적용 |
| 공동사용자 | 대표자가 초대한 보관함 공유자. 상한: M=2, L=3, XL=4 (대표 포함) |
| 보증금 반환 토큰 | 만료 당일 발급되는 7일 유효 URL 토큰. 고객이 반환 계좌 정보 등록 |

### 1.7 기술 스택 — 3개 옵션 비교

| 항목 | 옵션 1 (권장) | 옵션 2 | 옵션 3 (비권장) |
|------|-------------|--------|----------------|
| 프론트/백 | **Next.js 14 App Router + TS** | Next.js 프론트 + FastAPI(Python) 백엔드 분리 | Remix or Hono + Cloudflare Pages |
| DB | **Supabase (Postgres + Auth + Storage + Realtime)** | Neon/Railway Postgres + 자체 Auth(JWT) | Cloudflare D1 (SQLite 기반) |
| ORM | Drizzle ORM | SQLAlchemy(Py) + Drizzle(TS) | Drizzle ORM |
| 배포 | **Vercel** | Vercel(프론트) + Railway(백) | Cloudflare Pages + Workers |
| 배치 | Vercel Cron + Supabase Scheduled Functions | Railway Cron | Cloudflare Cron Triggers |
| 토스 위젯 | React SDK 네이티브 ◎ | 동일(프론트 Next) ◎ | Hono/Remix에서 React 사용 가능 △ |
| eh.lee 친화도 | TS 학습 곡선 有, 단 AI 보조로 상쇄 | Python 강점 활용 ◎, 풀스택 복잡도 ▲ | Workers 경험 有 ◎, D1 제약 ▲ |
| 월 비용(예상) | $0~20 (Vercel Hobby + Supabase Free) | $10~30 (2곳 관리) | $0~5 (가장 저렴) |
| 운영 복잡도 | 낮음 (단일 레포, 단일 콘솔) | 중간 (2-tier) | 중간 (D1 제약 회피 로직) |

**권장: 옵션 1 (Next.js + Supabase + Vercel)**

- 토스페이먼츠 결제 위젯이 React 네이티브 → 연동 리스크 최소
- Supabase 단일 콘솔에서 DB/Auth/Storage 관리 → 1인 운영에 최적
- Next.js Server Actions + Route Handlers 로 풀스택 단일 레포 → AI 보조 개발 시 컨텍스트 스위칭 최소
- Drizzle + TS 타입 안전성 → 스키마 변경이 코드에 즉시 반영, 회귀 오류 방지
- Vercel 무중단 배포 + PR 프리뷰 → 스테이징/프로덕션 분리 간단

**차선: 옵션 2** — Python/SQL 강점을 백엔드 주 언어로 쓰고 싶거나 나중에 분석/ML 워크로드가 붙을 가능성이 크면 선택.

**비권장: 옵션 3** — D1의 복잡 쿼리/동시성 한계, 토스 위젯 SSR 처리 제약, 풀스택 프레임워크 성숙도가 옵션 1 대비 열세. 단, 비용이 절대적 제약이면 재검토.

> **사용자 결정 필요**: 옵션 1/2/3 중 최종 선택. 이하 본 설계서는 **옵션 1** 기준으로 기술한다.

### 2.2 Phase별 상세

각 Phase는 `{목표, 단계, LLM 판단 vs 코드 처리, 성공 기준, 검증 방법, 실패 처리}` 구조로 정의한다.

#### Phase 0. 프로젝트 준비

| 항목 | 내용 |
|------|------|
| 목표 | 개발 환경·레포·시크릿 준비, 스택 확정 |
| 단계 | 0.1 스택 결정 → 0.2 Next.js/Supabase/Drizzle 초기 셋업 → 0.3 env 수집 |
| LLM 판단 | 스택 옵션 간 트레이드오프 요약, eh.lee 기술 수준 고려한 권장 |
| 코드 처리 | `npx create-next-app`, Supabase 프로젝트 생성 CLI, `.env.local` 스캐폴드 |
| 성공 기준 | `pnpm dev` 기동 OK, Supabase 연결 OK, TS 타입체크 통과 |
| 검증 방법 | 규칙 기반(빌드/타입체크/헬스 엔드포인트 200) |
| 실패 처리 | 자동 재시도(의존성 이슈, 최대 2회) → 에스컬레이션(버전 충돌/권한) |

#### Phase 1. 도메인 모델

| 항목 | 내용 |
|------|------|
| 목표 | Google Sheets 구조 → Postgres 스키마 이관 설계 확정 |

| 검증 방법 | 스키마 검증(매핑 매트릭스 체크리스트) + LLM 자기 검증(누락/모순) + **사람 검토**(확정) |
| 실패 처리 | 매핑 누락 시 자동 재시도(1회), 사용자 리뷰에서 반려 시 에스컬레이션 |

#### Phase 2. 비즈니스 로직 이관

| 항목 | 내용 |
|------|------|

### 2.3 상태 전이 (Phase 간)

- Phase N 완료 조건을 만족해야 Phase N+1 시작. 완료 상태는 `output/phase-N/complete.json`에 기록.
- Phase 1(스키마 확정), Phase 6(마이그레이션 실행/프로덕션 배포)은 **인간 승인 게이트**. 메인 에이전트가 자동 진행 금지.
- Phase 2 ~ 5는 병렬화 가능한 하위 작업 있음(예: 2.2 가격, 2.3 상태 머신 동시 진행). 단, Phase 3/4는 Phase 1 스키마 확정 후 시작.

## 4. 데이터 전달 및 상태 관리

### 4.1 중간 산출물 저장 위치

모든 Phase 간 데이터는 `output/` 하위에 파일로 저장. 이유: (a) 컨텍스트 토큰 절약, (b) Phase 재실행 시 이전 결과 참조, (c) 감사 추적.

### 4.2 파일 기반 vs 인라인 기준

| 사례 | 방식 |
|------|------|
| 스키마 매핑 매트릭스(수십~수백 행) | 파일 |
| GAS 기능 인벤토리 | 파일 |
| 알림톡 템플릿 매핑(13행) | 파일 + 요약 인라인 |
| Phase 간 완료 플래그 + 요약 | 파일 (`complete.json`) |
| 단일 쿼리 결과 요약 | 인라인 |
| 단순 의사결정(옵션 중 1개) | 인라인 |

### 4.3 진행 상태 추적 규약

- Phase 시작 시 `output/phase-{n}/start.json` 작성(start_at, input_artifacts).
- Phase 완료 시 `output/phase-{n}/complete.json` 작성(completed_at, artifacts, human_approval, validation_results).
- 메인은 세션 시작 시 `output/phase-*/complete.json`을 훑어 현재 Phase 판단.
- 중단되었다면 가장 최신 `start.json` 있고 `complete.json` 없는 Phase부터 재개.

> - Playwright E2E 핵심 플로우 통과
> - 보안 규칙(시크릿 노출, XSS, SQL injection) 자동 검사
