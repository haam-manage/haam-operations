## 2. 현재 고객 여정 (AS-IS)

### 전체 플로우

```
네이버 지도 검색 "셀프스토리지" / "개인창고"
  ↓
네이버 예약 (방문 예약 또는 계약 예약)
  ↓
관리자(eh.lee)가 네이버 예약 스크린샷을 텔레그램 봇에 전송
  ↓
Gemini AI가 이미지 OCR → 계약 정보 자동 추출
  ↓
Google Sheets(Active)에 자동 기록
  ↓
카카오 알림톡으로 고객에게 예약 확정 + 결제 안내 자동 발송
  ↓
고객 입금 (IBK 기업은행 계좌이체) → 관리자가 "완료" 스크린샷 전송
  ↓
결제 완료 알림톡 + QR 등록 대기열 등록
  ↓
매장 PC에서 QR봇(pywinauto)이 Hugen-V3 시스템에 자동 등록
  ↓
고객이 QR코드로 보관함 출입
```

### 핵심 문제점 (Pain Points)

1. **네이버 예약 의존**: 네이버 예약이 유일한 접수 채널. 네이버 수수료 발생, 예약 폼 커스터마이징 불가
2. **수동 결제 처리**: 고객이 IBK 계좌로 직접 이체 → 관리자가 확인 → 텔레그램에 완료 스크린샷 전송. 자동화 불가
3. **이중 작업**: 네이버 예약 화면을 스크린샷 찍어 텔레그램에 보내는 수동 단계 존재
4. **토스페이먼츠 미적용**: LinkPay API 코드는 완성했지만 아직 심사 중. 방향 전환 예정
5. **웹사이트 부재**: 독립적인 예약/결제/고객관리 웹사이트가 없음
6. **모바일 미최적화**: 관리 대시보드(GAS WebApp)는 기본적인 기능만 존재

## 3. 현재 시스템 아키텍처

### 기술 스택

| 구성요소 | 기술 | 역할 |
|---------|------|------|
| 서버 | Google Apps Script (GAS) | 전체 백엔드 로직 |
| 데이터베이스 | Google Sheets | 고객, 계약, 로그 등 모든 데이터 |

| 관리자 인터페이스 | Telegram Bot | 계약 처리, 명령어 실행 |
| AI | Gemini 2.5 Flash | 예약 스크린샷 OCR |
| 프론트엔드 | Cloudflare Workers (정적) | 공동사용자 등록, 보증금반환 계좌 등록 |
| QR 등록 | Python (pywinauto) | 매장 PC에서 Hugen-V3 자동 조작 |
| 일정 | Google Calendar | 방문 예약 일정 관리 |

### Google Sheets 구조

#### Active 시트 (핵심 — 현재 계약 고객)

| 열 | 필드 | 설명 | 예시 |
|----|------|------|------|
| A | ID | 고객 ID (보관함-전화뒷4) | M01-1234 |
| B | NAME | 성명 | 이은호 |
| C | PHONE | 연락처 (텍스트 형식) | '01012341234 |
| D | CABINET | 보관함 번호 | M01, L03, XL02 |
| E | START | 계약 시작일 | 2026-03-15 |
| F | MONTHS | 이용 기간 (개월) | 3 |
| G | EXPIRY | 계약 만료일 | 2026-06-15 |
| H | REMAINING | 남은 일수 | =G-TODAY() (수식) |
| I | STATUS | 결제 상태 | "완료" / "미납" / "방문" |
| J | RENTAL | 렌탈료 (원) | 140000 |
| K | DEPOSIT | 보증금 (원) | 70000 |
| L | SECURE | 보안코드 (5자리) | 15234 |
| M | RENEWAL | 갱신/연체 상태 | (아래 상태 머신 참조) |
| N | REMARK | 비고 | M01 누적 3개월 이용 |
| O | CO_USERS | 공동사용자 | 홍길동(01098765432) |

#### RENEWAL 상태 머신 (M열)

```
(비어있음) → 정상 이용 중
  ↓ D-7
'안내완료(D-7)' → 만료 7일 전 알림 발송 완료
  ↓ D-0
'종료' → 계약 만료 당일 처리 완료 (보증금 반환 프로세스 시작)
  ↓ D+3
자동 아카이빙 → Archive 시트로 이동

'연체관리' → 관리자가 수동 지정 (만료 후에도 물건을 안 빼간 경우)
  → '연체(D+1)' → '연체(D+3)' → '연체(D+7)' 순차 에스컬레이션

'재계약' → 기존 고객 재계약 (D-7, D-0 알림 스킵)
```

#### 기타 시트 목록

| 시트명 | 용도 |
|--------|------|
| Archive | 만료/종료 고객 보관 |
| SystemLog | 통합 로그 (72시간 자동 정리) |
| TaskQueue | 작업 큐 (신규계약, 재계약, 결제완료, QR등록) |
| 공동사용자 | 등록된 공동사용자 목록 |
| 공동사용자초대대기 | 초대 토큰 (24시간 유효) |
| QR등록대기 | QR봇이 감시하는 대기열 |
| 금액불일치 | Gemini 인식 금액 ≠ 계산 금액 시 대기 |
| 보증금반환계좌 | 반환 토큰 (7일 유효) + 은행 정보 |
| Gemini재시도대기 | AI 분석 실패 시 재시도 큐 |
| 공지 템플릿 | 일괄 발송용 템플릿 |
| 발송 로그 | 알림톡 발송 이력 |

## 4. 핵심 워크플로우 상세

### 4-1. 계약 처리 (신규/재계약)

```
[관리자] 네이버 예약 스크린샷 → 텔레그램 봇 전송
  ↓
[시스템] Gemini AI OCR 분석 (3회 재시도, 503→10초/429→60초 대기)
  추출 정보: 이름, 전화번호, 보관함, 개월수, 프로모션 여부, 시작일, 금액, 뱃지(확정/완료)
  ↓
[분기] 뱃지 = "확정" (예약만 한 상태)
  ├─ Active 시트에 기록 (신규: appendRow / 재계약: 기존 행 업데이트)
  ├─ 가격 계산 (프로모션 ON이면 프로모 가격, OFF면 일반 할인)
  ├─ 금액 검증: Gemini 인식 금액 vs 시스템 계산 금액
  │  ├─ 일치 → 알림톡 발송 (예약확정 + 결제안내) + IBK 입금 정보 관리자 알림
  │  └─ 불일치 → 금액불일치 시트에 저장 → 관리자 "금액확정" 명령 대기
  └─ QR등록대기에 등록

[분기] 뱃지 = "완료" (결제까지 완료된 상태)
  ├─ Active 시트 STATUS → "완료"
  ├─ 결제완료 알림톡 발송 (보안코드 포함)
  └─ QR등록대기에 등록
```

### 4-2. 결제 처리 (현재)

**현재 방식**: 고객이 IBK 기업은행으로 직접 계좌이체

```
알림톡으로 결제 안내 수신 (은행명, 계좌번호, 금액)
  ↓
고객이 직접 계좌이체
  ↓
관리자가 입금 확인 후 네이버에서 "결제완료" 처리
  ↓
"완료" 스크린샷을 텔레그램에 전송
  ↓
시스템이 자동으로 Active STATUS 갱신 + 결제완료 알림톡 발송
```

> **새 웹사이트에서**: 토스페이먼츠 결제 위젯을 통해 온라인 결제로 전환 예정. 계좌이체 수동 확인 프로세스 제거.

### 4-3. 만료/연체 자동화 (매일 09:00 KST)

```
[09:00 자동 실행]

1단계: 미납 정리
  STATUS='미납' AND 시작일+2일 경과 → 행 삭제 + 관리자 알림

2단계: 아카이빙
  RENEWAL='종료' AND 남은일수 < -2 → Archive로 복사 → Active에서 삭제

3단계: 만료 알림 (STATUS='완료'인 행만)
  D-7: RENEWAL이 비어있으면 → 만료예고 알림톡 → RENEWAL='안내완료(D-7)'
  D-0: RENEWAL≠'재계약','종료' → 보증금반환 토큰 생성 → 만료 알림톡(반환 링크 포함) → RENEWAL='종료'

4단계: 연체 에스컬레이션 (RENEWAL='연체관리' 이후)
  D+1 → 연체 알림 → '연체(D+1)'
  D+3 → 연체 알림 → '연체(D+3)'
  D+7 → 최종 경고 → '연체(D+7)'

5단계: 로그 정리 (72시간 이상 된 로그 삭제)
```

### 4-4. 보증금 반환

```
[자동] 만료 당일(D-0) 알림톡에 반환 계좌 등록 링크 포함
  URL: https://haam-register.haam-manage.workers.dev/refund/?refund_token=TOKEN
  ↓
[고객] 링크 접속 → 은행명, 계좌번호, 예금주 입력 → 제출
  ↓
[시스템] 보증금반환계좌 시트에 저장, 상태='반환대기', 관리자 텔레그램 알림
  ↓
[관리자] 수동으로 실제 계좌이체 후 시트에서 상태='반환완료'로 변경
  ↓
[트리거] onEdit → 처리완료 시각 자동 기록
```

### 4-5. 공동사용자 등록

```
[관리자] 대표자 이름, 전화번호, 보관함 번호로 초대 생성
  ↓
[시스템] 초대 토큰 생성 (24시간 유효) → 초대 알림톡 발송
  URL: https://haam-register.haam-manage.workers.dev/?token=TOKEN
  ↓
[공동사용자] 링크 접속 → 이름, 전화번호, 약관 동의 → 제출
  ↓
[시스템]
  - 보안코드 자동 생성 (5자리)
  - 공동사용자 시트에 등록
  - QR등록대기에 추가
  - 공동사용자에게 등록완료 알림톡 (보안코드 포함)
  - 대표자에게 확인 알림톡

[용량 제한] M=2명, L=3명, XL=4명 (대표자 포함)
```

## 5. 외부 서비스 연동 현황

### 카카오 알림톡 (Solapi)

등록된 템플릿 13개:

| 템플릿 | 발송 시점 | 주요 변수 |
|--------|----------|----------|

| 보관함 예약확정 | 계약 예약 처리 시 | 고객명, 보관함, 시작/종료일, 보증금, 렌탈, 총액 |

| 개월변경 확정 | 개월 변경 시 | 변경 전/후 개월, 추가결제금액 |
| 공동사용자 초대 | 관리자가 초대 생성 시 | 공동이용자, 보관함, 초대링크 |
| 공동사용자 등록완료 | 공동사용자 등록 완료 시 | 보안코드 포함 |
| 대표자 확인 | 공동사용자 등록 시 대표자에게 | 공동사용자명, 연락처 |
| 보관함번호 오류 | 보관함 번호 이상 시 | 고객명 |
| 만료 D-7 | 계약 종료 7일 전 | 보관함, 종료일 |
| 만료 D-0 | 계약 종료 당일 | 보관함, 종료일, 반환계좌 링크 |
| 연체 D+1 | 연체 1일 | (미등록 — 'x') |
| 연체 D+3 | 연체 3일 | (미등록 — 'x') |
| 연체 D+7 | 연체 7일 | (미등록 — 'x') |

### 텔레그램 봇
- 1분 주기 폴링 (LockService로 동시 실행 방지)
- 이미지 수신 → Gemini AI OCR → 자동 처리
- 텍스트 명령어 처리 (금액확정, 개월변경)
- 공지 일괄발송 큐 처리

### Google Calendar
- 방문 예약 시 자동으로 캘린더 일정 생성

### Hugen-V3 (QR 시스템)
- 매장 물리 PC에서 pywinauto 봇이 QR등록대기 시트를 감시
- 대기열에 항목이 있으면 Hugen-V3 관리 프로그램에 자동 입력
- 이 부분은 웹사이트 전환과 무관하게 유지 (물리 장비 연동)

## 6. 토스페이먼츠 현황

### 현재 상태

- **결제 수단**: 카드결제, 계좌이체, 가상계좌 등 (심사 통과 후 사용 가능)
- **연동 방식 (새 계획)**: 토스페이먼츠 결제 위젯 (JavaScript SDK) → 웹사이트 내 결제 페이지에 삽입

## 7. 새 웹사이트 요구사항 (TO-BE)

### 핵심 방향

**네이버 지도 → 웹사이트 → 결제(토스페이먼츠)** 이 하나의 흐름으로 완결되어야 한다.

네이버 예약을 완전히 대체한다. 네이버 지도에서 유입되는 트래픽이 네이버 예약이 아닌 자체 웹사이트로 향하도록 한다.

### 필수 기능

1. **예약/계약 페이지** (모바일 최적화)
   - 보관함 사이즈 선택 (M/L/XL)
   - 이용 기간 선택 (1/3/6/12개월)
   - 자동 가격 계산 (프로모션 적용 포함)
   - 고객 정보 입력 (이름, 전화번호)
   - 이용 시작일 선택
   - 토스페이먼츠 결제 위젯으로 즉시 결제

2. **결제 시스템**
   - 토스페이먼츠 결제 위젯 (카드, 계좌이체 등)
   - 결제 완료 후 자동으로 계약 확정
   - 웹훅으로 결제 상태 실시간 반영

3. **고객 마이페이지** (있으면 좋고, 필수는 아닐 수 있음)
   - 계약 현황 조회
   - 공동사용자 초대
   - 보증금 반환 계좌 등록
   - 재계약/연장

4. **관리자 대시보드**
   - 현재 계약 현황
   - 매출/결제 현황
   - 만료 예정 고객 목록
   - 수동 작업 (상태 변경, 메모 등)

5. **알림 시스템**
   - 기존 카카오 알림톡 유지 (Solapi)
   - 결제 완료, 만료 안내, 연체 경고 등 기존 알림 플로우 이관

### 유지해야 할 것

- 카카오 알림톡 (Solapi) — 고객 소통의 핵심
- QR 등록 프로세스 — Hugen-V3 물리 장비 연동은 유지
- 만료/연체 자동화 로직 — 비즈니스 룰 동일
- 가격 정책 및 프로모션 구조 — 그대로 이관
- 보증금 반환 프로세스 — 웹사이트 내로 통합
- 공동사용자 시스템 — 웹사이트 내로 통합

### 제거/대체할 것

- 네이버 예약 의존성 → 자체 예약 페이지
- 텔레그램 봇 이미지 처리 → 웹사이트에서 직접 계약 → Gemini OCR 불필요
- IBK 수동 계좌이체 → 토스페이먼츠 온라인 결제
- Google Sheets 데이터베이스 → 적절한 DB (설계 시 결정)
- GAS 백엔드 → 웹 프레임워크 (설계 시 결정)

### 기존 프로세스 중 재설계 필요

1. **계약 플로우**: 텔레그램 이미지→OCR 대신, 고객이 직접 웹에서 입력+결제
2. **결제 확인**: 수동 스크린샷 대신, 토스페이먼츠 웹훅 자동 처리
3. **관리자 인터페이스**: 텔레그램 명령어 대신, 웹 관리자 대시보드
4. **데이터 저장**: Google Sheets 대신 DB (기존 시트 구조를 DB 스키마로 변환)

## 8. 비즈니스 규칙 요약 (반드시 새 시스템에 반영)

### 가격 계산 로직

```
IF 프로모션 활성:
  1개월 → 정가 × 1 × (1 - 15%) = M 기준 ₩59,500
  3개월 → 정가 × 2 (1개월 무료) = M 기준 ₩140,000
  6개월 → 정가 × 4 (2개월 무료) = M 기준 ₩280,000
  12개월 → 정가 × 8 (4개월 무료) = M 기준 ₩560,000
ELSE:
  총액 = 정가 × 개월수 × (1 - 할인율)
  할인율: 12개월↑=20%, 6~11=15%, 3~5=10%, 1~2=0%

보증금은 항상 별도: M=₩70,000 / L=₩120,000 / XL=₩200,000
총결제금액 = 렌탈료 + 보증금
```

### 만료 프로세스 타임라인

```
D-7  : 만료 예고 알림
D-0  : 계약 종료 + 보증금 반환 안내
D+3  : 미반출 시 자동 아카이빙
D+1  : 연체 1차 경고 (관리자 수동 지정 후)
D+3  : 연체 2차 경고
D+7  : 연체 최종 경고
```

### 미납 처리

```
예약 후 2일 내 미입금 → 자동 삭제 (예약 취소)
```

### 보안코드

```
5자리 숫자 (첫자리 1~9, 나머지 0~9 랜덤)
계약 완료 시 또는 공동사용자 등록 시 자동 생성
QR 출입에 사용 (Hugen-V3)
```

### 공동사용자 제한

```
M: 최대 2명 (대표 1 + 공동 1)
L: 최대 3명 (대표 1 + 공동 2)
XL: 최대 4명 (대표 1 + 공동 3)
```
